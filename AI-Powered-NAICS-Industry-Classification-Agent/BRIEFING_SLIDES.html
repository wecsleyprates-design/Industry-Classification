Executive Summary

Worth AI Today — 3 Key Facts

❌ No ML model in classification. XGBoost only exists in entity matching. The classification decision is a deterministic rule (factWithHighestConfidence()), not a trained model.
❌ UK SIC is silently discarded. OpenCorporates returns "us_naics-541110|uk_sic-62012". The loop only reads us_naics. The UK SIC sits in the database, never used.
❌ One taxonomy, one code. Every company gets US NAICS only. UK/EU/global companies receive an inappropriate US code or naics_id = NULL.

Consensus Engine — What Changes

✅ Real ML for classification. XGBClassifier with 38 engineered features outputs calibrated probabilities over all codes — not a rule.
✅ 6 taxonomies simultaneously. NAICS 2022, UK SIC 2007, NACE Rev.2, ISIC Rev.4, US SIC 1987, MCC — correct taxonomy auto-selected per jurisdiction.
✅ AML/KYB risk signals. 9 signal types (shell company, registry discrepancy, temporal pivot, source conflict…). Worth AI produces zero.

The Worth AI Pipeline — How It Actually Works Today

The Two XGBoost Models in Worth AI

Model 1 ✅ EXISTS — Entity Matching
File: entity_matching/core/matchers/matching_v1.py
Model: entity_matching_20250127
Question it answers: "Is this database record the same company as the input?"
Output: match_confidence 0.0–1.0 per candidate
Threshold: ≥ 0.80 = MATCHED

Uses 26 name+address similarity features:
Jaccard k-gram (k=1–4), word-level Jaccard,
postal code match, city match, street number match
Model 2 ❌ DOES NOT EXIST — Classification
There is no ML model for choosing which NAICS code to assign.

The classification decision is made by a deterministic rule:

factWithHighestConfidence()
winner = argmax(source.confidence × weight)

This rule:
• Cannot learn from feedback
• Cannot detect conflicts systematically
• Cannot output probabilities
• Cannot route by jurisdiction

The Fact Resolution Rule — Not ML

How the winner is chosen
factWithHighestConfidence():
  winner = argmax(
    source.confidence × source.weight
  )
  if two scores differ by < 0.05:
    use weightedFactSelector(
      fact_level_weight
    )
What this cannot do:
• Learn from manual overrides
• Output a probability (always 1 winner, no alternatives)
• Detect when sources conflict systematically
• Route to different taxonomy by jurisdiction

The 4 Confirmed UK Classification Gaps

Result: Every UK business either gets naics_id = NULL or an inappropriate US NAICS code applied to it.

Gap 1 — OpenCorporates resolver
Returns "us_naics-541110|uk_sic-62012". Loop filter: codeName.includes("us_naics") — UK SIC parsed into classification_codes fact. That fact has zero downstream consumers. Data is in the database, never used.
Gap 2 — Trulioo .sicCode never read
The object {'{ naicsCode: "...", sicCode: "62012" }'} is in the API response. The fact engine reads ?.naicsCode and never accesses .sicCode. One missing field access.
Gap 3 — No persistence layer
Zero migrations create core_uk_sic_code table or add uk_sic_id to data_businesses. Even if both gaps above were fixed, the data has nowhere to go.
Gap 4 — AI enrichment Zod schema
uk_sic_code field not in the Zod schema for AI enrichment. Even when the AI returns it for GB companies, it cannot be persisted. Structurally impossible without a schema change AND a new table.

The AI Enrichment Step — Worth AI

Model: GPT-5-mini (gpt-5-mini)
Via openai.responses.create with Zod schema enforcement
When it triggers (conditional)
Runs when: website + name + address resolved AND 1–2 NAICS sources
Skips when: already ≥ 3 NAICS sources (saves credits)
Defers: Bull queue, 10 attempts, exponential backoff
Force-runs: after 3-minute timeout
Confidence → fact weight
HIGH → 0.20  |  MED → 0.15  |  LOW → 0.10
Even "HIGH" loses to OpenCorporates (0.90)
Last resort: If NAICS not in core_naics_code, replaced with hardcoded 561499 (Administrative Services)

External Registry Lookups — New in the Consensus Engine

Before the LLM is called, the engine queries free government registries for authoritative codes the company filed directly with the government.

SEC EDGAR — US companies
Free — no API key required
Endpoint: data.sec.gov/submissions/CIK{cik}.json
Search: efts.sec.gov/LATEST/search-index
Returns: SIC code + description, entity type, state of incorporation, ticker, former names
Coverage: all US public SEC filers

Apple Inc example (live verified):
"SEC EDGAR filing (AAPL): SIC 3571 — Electronic Computers, incorporated in CA. Entity type: operating."

This is the SIC code Apple told the US government. Ground truth.
Companies House — UK companies
Free API key from developer.company-information.service.gov.uk
Returns: up to 4 SIC codes, company type, status, date of creation
Coverage: all UK registered companies

Barclays example:
"Companies House: SIC codes [64191 — Banks, 64992 — Factoring]. Type: plc, Status: active."

Set COMPANIES_HOUSE_API_KEY env var.
Degrades gracefully without key.
⚠️ Worth AI has NO equivalent — it never queries any government registry directly for classification

The Improved AI Prompt — LLM as Referee

Worth AI — Blind Classifier

LLM receives only:
• Company name + address
• Web search text snippet
• No vendor signal context
• No registry data

LLM guesses from scratch. Even if OpenCorporates already has the answer, the LLM doesn't know.
Trigger: conditional (1–2 sources, async Bull queue)
Weight: 0.10 (always loses to vendors)
Last resort: hardcoded 561499

Consensus Engine — Evidence Referee

LLM receives 5 evidence blocks:
1. Registry data (SEC EDGAR / Companies House) — ground truth
2. Vendor signals: what all 6 sources returned with weight + status
3. Source agreement summary ("4/5 agree on code 334118")
4. UGO FAISS semantic candidates (top-8 per taxonomy)
5. 5-step reasoning instruction
Reasoning order:
Step 1: Registry first (self-reported, highest authority)
Step 2: Vendor consensus (confirm if 3+ agree)
Step 3: Adjudicate conflicts (explain divergence)
Step 4: Select from UGO candidates
Step 5: MCC + AML risk flag
New output fields: source_used (registry/vendor_consensus/semantic/web_inference) · registry_conflict (bool) · mcc_risk_note

The Unified Global Ontology (UGO)

A FAISS cosine vector index (IndexFlatIP) built from all 2,330 taxonomy codes embedded with all-MiniLM-L6-v2 (384-dim). Enables Cross-Ontology Embedding Alignment — no hardcoded crosswalk tables needed.

Search example: "licensed restaurant food service"
→ UK SIC 56101 "Licensed restaurants" (sim: 0.741) ← top result
→ NAICS 722511 "Full-Service Restaurants" (sim: 0.672)
→ ISIC 5610 "Restaurant and mobile food" (sim: 0.643)
Cross-taxonomy distance:
UK SIC 56101 ↔ NAICS 722511 = 0.09 (nearly identical meaning)
Restaurant code ↔ Banking code = 0.89 (completely different)
