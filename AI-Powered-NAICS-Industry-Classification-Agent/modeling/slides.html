The Production Rule (customer_table.sql)

-- From sp_recreate_customer_files()
primary_naics_code = CASE
    WHEN zi_match_confidence > efx_match_confidence
        THEN zi_c_naics6           -- ZoomInfo wins
    ELSE
        efx_primnaicscode          -- Equifax wins
END

✅ What gets stored

  • Winning NAICS code → customer_files.naics_code
  • Match confidences → zi_match_confidence, efx_match_confidence
  • OC uid field (raw) → industry_code_uids

❌ What never gets stored

  • UK SIC from OC — no table exists
  • Probability — never computed
  • Alternative codes — single code only
  • AML signals — no risk engine
  • KYB recommendation — not produced
  • OC / Liberty industry codes — never routed

Why OC and Liberty Never Appear as Winning Source

The production rule only compares two sources:

IF zi_confidence > efx_confidence → ZoomInfo wins
ELSE                              → Equifax wins

OpenCorporates and Liberty are used by Level 1 — their match confidences are computed and stored. But the production classification rule never reads their industry codes.

OpenCorporates industry_code_uids contains richer data than ZoomInfo or Equifax — it includes UK SIC, NACE, NAICS, and more in a single pipe-delimited field. None of it is used by the production rule.

This is confirmed by reading customer_table.sql in the SIC-UK-Codes repo.

In the Consensus model, all 4 sources (OC, EFX, ZI, Liberty) contribute as weighted features to Level 2.

Consensus Columns — Primary Code and Probability

Cons: Primary code

Top-1 prediction from Level 2 XGBoost. Scores all possible NAICS codes simultaneously using all 6 vendor signals.

Correct for LAKEWOOD HOTEL: 721110 (Hotels) vs production's 561320 (Staffing).

Cons: Primary taxonomy

Jurisdiction-aware taxonomy routing:

  • US/CA/AU → NAICS 2022
  • GB/GG/JE → UK SIC 2007
  • EU → NACE Rev.2
  • Global → ISIC Rev.4

Production always returns NAICS.

Cons: Probability

Calibrated confidence from multi:softprob output:

  • ≥ 70% → High confidence → APPROVE
  • 40–69% → Medium → REVIEW
  • < 40% → Low → LOW_CONSENSUS_PROB flag

Production probability: None.

Cons: Secondary codes

2nd and 3rd most probable codes. Wide spread between primary and secondary = hybrid business or unreliable vendor data signal.

Consensus Columns — AML Signal Features

ColumnWhat it measuresAML flag threshold
Cons: Majority agreement Fraction of sources returning the same 6-digit code (0–1) < 0.40 → SOURCE_CONFLICT
Cons: Temporal pivot Rate of industry code change across last 3 API calls > 0.50 → STRUCTURE_CHANGE
Cons: Web↔Registry dist Semantic distance between web presence and OC registry label (0=same, 1=completely different) > 0.55 → REGISTRY_DISCREPANCY
Cons: Sources matched Count of sources with Level 1 confidence ≥ 0.80 0 = no confident match anywhere
Cons: Risk score Composite 0–1 score from all signals ≥ 0.75 = REJECT
Cons: KYB recommendation APPROVE / REVIEW / ESCALATE / REJECT Production: never produced
All of these signals come from data already in Redshift. The feature engineering reads Level 1 confidence scores, vendor industry codes, and OC jurisdiction codes — all stored from the original API calls.

FlagWhat triggers itRisk weight

REGISTRY_DISCREPANCY Web↔Registry semantic distance > 0.55 25%

LOW_CONSENSUS_PROB Level 2 top-1 probability < 40% — model is uncertain 10%

Production fires zero AML flags. It has no risk engine.

B — Match Flags (6) binary ≥ 0.80

oc_matched, efx_matched, zi_matched, liberty_matched, tru_matched, ai_matched

C — Code Signals (6) AML features

tru_pollution_flag, web_registry_distance, temporal_pivot_score, cross_taxonomy_agreement, source_majority_agreement, source_code_diversity

D — Jurisdiction Flags (12) from OC jurisdiction_code

j_us, j_us_state, j_ca, j_ca_province, j_eu, j_apac, j_latam, j_mena, j_afr, j_other, is_subnational, is_naics_jurisdiction

is_holding, is_ngo, is_partnership

F — Aggregate Quality (4)

hi_risk_sector, avg_confidence, max_confidence, sources_above_threshold

All 45 features derived from data already in Redshift. No new API calls required.

How to Get Real Results in the Notebook

Current status (without VPN / public access)

Redshift is VPC-private. Notebook uses synthetic data that mirrors the exact Redshift schema with realistic confidence distributions.

Synthetic data uses the same column names, same code ranges, same confidence distributions as real data.

To use real Redshift data

Option A: Enable Redshift public access
AWS Console → Redshift Serverless → Workgroup → Edit → Toggle "Publicly accessible" ON

Option B: Run from inside the VPC (EC2 or VPN)

Then set env vars and delete experiment_results.csv to regenerate.

export REDSHIFT_HOST=worthai-services-redshift-endpoint-...
export REDSHIFT_USER=readonly_all_access
export REDSHIFT_PASSWORD=...
export REDSHIFT_DB=dev
rm experiment_results.csv
jupyter notebook experiment_notebook.ipynb
