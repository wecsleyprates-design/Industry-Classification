The Most Important Insight — Before Any Phase

Worth AI already receives all the data it needs for complete global industry classification.
The pipeline collects it, stores a fraction of it, and discards the rest at Phase 4.

DATA ALREADY IN THE PIPELINE — NOT USED

uk_sic-62012 in OpenCorporates industry_code_uids — loop only reads us_naics
Trulioo returns { naicsCode, sicCode } — .sicCode is adjacent, never read
Equifax: 25 industry columns available — only primnaicscode used (24 dropped)
jurisdiction_code for 16M+ GB companies in Redshift — used for matching, not taxonomy routing
rel_business_industry_naics has historical codes — never checked for pivot/change patterns
match_confidence per source from Model 1 — used to pick 1 winner, never as 38-feature input

WHAT THE CONSENSUS ENGINE DOES WITH IT

Reads ALL entries from industry_code_uids → routes uk_sic to UK SIC 2007 output
Reads Trulioo .sicCode alongside .naicsCode — 2nd UK SIC source immediately
Uses Equifax secondary NAICS + SIC as additional taxonomy outputs
Routes jurisdiction_code → correct taxonomy: gb → UK SIC, de → NACE, us_mo → NAICS
Computes temporal pivot score from rel_business_industry_naics → STRUCTURE_CHANGE AML signal
Encodes all 6 match_confidence values as features 0–11 in Model 2's 38-feature vector

Where industry_code_uids comes from

Apple Inc registers with California Secretary of State
↓ CA assigns NAICS 334118 to Apple's business type
UK Companies House assigns SIC 26400 to Apple UK
↓ OpenCorporates scrapes both government registries
industry_code_uids = "us_naics-334118|uk_sic-26400|ca_naics-334118"
↓ Worth AI ingests into Redshift
Waiting in open_corporates_standard_ml_2 — millions of rows
Key insight: Worth AI doesn't compute industry codes — it reads them from government registries via OpenCorporates. Model 1's job is simply to find which pre-loaded row belongs to the current applicant.

3Phase 3 — Entity Matching XGBoost (Model 1)

Runs for Equifax, ZoomInfo, Liberty Data (bulk Redshift files). OC and Trulioo already returned the right company in Phase 2.

INPUT: company_name + address + country (from applicant)
↓
STEP 1: Canonise name
"Apple Inc" → strip "Inc" → uppercase → "APPLE"
↓
STEP 2: Block candidates from Redshift
WHERE LEFT(normalised_name, 3) = 'APP' AND region = 'CA'
→ ~200 candidates (not 10 million)
↓
STEP 3: Compute 26 similarity features per candidate pair
10 name + 10 address + 6 short name features
↓
STEP 4: XGBoost predict_proba(26_features)
"Apple Inc. Cupertino CA" → 0.97 ✅
"Apple Bank for Savings NY" → 0.12 ❌
↓
STEP 5: Threshold ≥ 0.80 → MATCHED
Pull industry fields from that row

Phase 3 does NOT choose between codes. It only matches records and reports match_confidence. The decision of which code wins happens in Phase 4.

Phase 4 — The Divergence Point

Same inputs enter Phase 4. Completely different algorithms. Completely different outputs.

Phase 4A — Worth AI: Deterministic Rule
factWithHighestConfidence()
winner = argmax(confidence × weight)

OC: 0.97 × 0.90 = 0.873 ← WINS
ZI: 0.94 × 0.80 = 0.752
EFX: 0.91 × 0.70 = 0.637
TRU: 0.62 × 0.70 = 0.434

Output: NAICS 334118 (no probability)
UK SIC: ❌ dropped
AML signals: ❌ none
Confidence: ❌ text string only
Phase 4B — Consensus Engine: 5-Step Stacking
Step A: 38-feature vector from all signals
Step B: XGBoost → P(334118)=0.92, P(541512)=0.04…
Step C: SEC EDGAR → SIC 3571 (ground truth)
Step D: LLM → NAICS + UK SIC + NACE + MCC
Step E: Risk Engine → 9 AML signals + KYB

Output: Top-5 codes with probabilities
UK SIC: ✅ 26400
AML signals: ✅ 9 types
Confidence: ✅ float 0–1 (calibrated)
No vendor contracts change. No Redshift tables change. No Kafka topics change. No entity matching model changes.
Only the decision algorithm in Phase 4 is replaced.

4APhase 4A — Worth AI: The Deterministic Rule in Detail

The Algorithm
factWithHighestConfidence():
  for each fact (naics_code, industry, mcc):
    candidates = all source values
    winner = argmax(confidence × weight)
    if diff < WEIGHT_THRESHOLD (0.05):
      use weightedFactSelector(fact_level_weight)

# For naics_code fact, Apple Inc:
OC: 0.97 × 0.90 = 0.873  ← WINNER
ZI: 0.94 × 0.80 = 0.752
EFX:0.91 × 0.70 = 0.637
TRU:0.62 × 0.70 = 0.434  ← CONFLICT silently ignored
SERP:0.75 × 0.30= 0.225
AI: 0.20 × 0.10 = 0.020
After the rule:
Kafka UPDATE_NAICS_CODE event
→ handleNaicsData() resolves naics_id from core_naics_code
→ UPDATE data_businesses SET naics_id=X, mcc_id=Y

7 Things the Rule Cannot Do

❌ No probability — always picks one winner, uncertainty invisible
❌ Cannot learn — manual overrides in rel_business_industry_naics ignored
❌ Conflict is silent — Trulioo says 541511, OC says 334118 → rule picks one, no flag
❌ Agreement ignored — 5/5 agree = same format as 1/5
❌ No taxonomy routing — GB company still gets NAICS, not UK SIC
❌ No AML signals — code changed 3× this year = treated as normal
❌ Single taxonomy only — cannot output NAICS + UK SIC + MCC simultaneously

4BPhase 4B — Consensus Engine: Steps A & B

Step A — 38-Feature Engineering
All vendor signals → numeric vector:

Features 0–11: Source quality
Weighted confidence per source (0.90×0.97=0.873…)
MATCHED flag per source (1 if ≥0.80)

Features 12–16: AML signals
Trulioo pollution flag · Web↔registry distance
Temporal pivot score · Cross-taxonomy agreement

Features 16–18: Entity type
Is Holding · Is NGO · Is Partnership

Features 19–30: Jurisdiction one-hot
US/US_STATE/CA/EU/APAC/LATAM/MENA/AFRICA…
Routes to correct taxonomy automatically

Features 31–37: Agreement & risk
Majority code agreement · High-risk NAICS prefix
Source count · Avg/max confidence
Step B — XGBoost Model 2 (multi:softprob)
# Input: 38-feature vector
# Output: probability distribution over all codes

probs = model.predict_proba(features)

# Apple Inc (5/5 sources agree, stable history):
P(NAICS 334118) = 0.92  ← PRIMARY ✅
P(NAICS 541512) = 0.04  ← secondary
P(NAICS 519290) = 0.02  ← secondary
P(NAICS 551112) = 0.01
P(NAICS 423690) = 0.01

# Unknown startup (sources conflict heavily):
P(NAICS 722511) = 0.14  ← LOW CONFIDENCE ⚠️
→ LOW_CONSENSUS_PROBABILITY risk signal fired
→ KYB: REVIEW (not APPROVE)
0.92 means: 92% of companies with similar 38-feature signatures were correctly classified at 334118. Actionable: threshold ≥ 0.70 → auto-approve.

4BPhase 4B — Consensus Engine: Steps C & D

Step C — External Registry Lookup (not in Worth AI)
Before the LLM runs, the engine queries free government registries:

SEC EDGAR (US companies, free, no key)
data.sec.gov/submissions/CIK0000320193.json
→ sic: "3571" — Electronic Computers (AAPL)
Apple itself reported this to the US government. Ground truth.

Companies House (UK companies, free key)
api.company-information.service.gov.uk/search/companies
→ sic_codes: ["26400"] — Consumer electronics

These are authoritative — the company told the government directly. Passed to LLM as the highest-priority evidence block.
Step D — LLM as Referee (GPT-4o-mini)
LLM receives 5 evidence blocks in priority order:

1. Registry data: "SEC EDGAR (AAPL): SIC 3571 — Electronic Computers"
2. Vendor signals: OC=334118 ✅ EFX=334118 ✅ TRU=541511 ⚠️ ZI=334118 ✅
3. Agreement: "4/5 sources agree on 334118"
4. UGO candidates: top-8 codes per taxonomy from FAISS
5. Reasoning instruction: Registry first → vendor consensus → adjudicate conflicts

LLM output — ALL taxonomies simultaneously:
primary: NAICS 334118 (US, high confidence)
uk_sic: 26400 (GB alternative — NOT dropped)
nace: C26 (EU alternative)
mcc: 5045 (Computers/peripherals — AML note: normal)

Summary


🟣 Phase 0: Industry codes pre-loaded from government registries via OC/Equifax/ZoomInfo/Liberty Data into Redshift. Already there before any applicant.
🟢 Phases 1–3: Applicant submits → live API calls → entity matching (Model 1) finds the right Redshift row per vendor. IDENTICAL in both systems.

🔵 Phase 4B (Consensus): 5-step stacking — 38 features → XGBoost probabilities → external registry → LLM referee → Risk Engine. Replaces only Phase 4.
