The Three Primary Identifiers

Every record is identified by three UUIDs forming a hierarchy:

Customer ID → Case ID → Business ID
🏦 Customer ID
The bank/lender using Worth AI.
Table: data_customers.id
URL: /customers/{customer_id}
Matches the warehouse ID — use in SQL to build direct links.
📋 Case ID
Point-in-time snapshot.
Table: data_cases.id
Primary use: auditability, traceability, compliance. Regulators require it.
"I look at case ID 0.1% of the time." (Allison)
🏢 Business ID
The merchant being verified.
Table: data_businesses.id
URL: admin.joinworth.com/{business_id}
Primary ID for 99.9% of analysis. Stable across all cases.
Craig (00:30:20): "The three IDs that drive the business are those three IDs."  |  Raj (00:33:18): "Case ID is for auditability, traceability. HSBC was asked to shut down for not having proper case management. TD Bank had a similar scenario."

What Is a Case? When Is a New One Created?

Definition (Allison, 00:09:00): "A case is a point in time of onboarding or risk monitoring — just pulling all of the verifications at a given point in time."

What creates a new case

  • Worth Score manual refresh — by analyst or business owner
  • Risk monitoring — flags a score change on existing business
  • New customer invitation — same business applies to different lender
Craig (00:35:48): Score refresh creates a new case but does NOT necessarily re-run all vendor integrations. Some facts may carry over.

Case types

TypeIDWhen
ONBOARDING1New business application (most common today)
APPLICATION_EDIT2Business updated their application

Status flow

SUBMITTED → [integrations run] → SCORE_CALCULATED → AUTO_APPROVED (6) if score ≥ threshold → AUTO_REJECTED (13) if score ≤ threshold → UNDER_MANUAL_REVIEW (4) if in middle range → MANUALLY_APPROVED (5) | MANUALLY_REJECTED (8) by analyst

Worth Score — How 819/850 Is Computed

Common misconception: The Worth Score is NOT the XGBoost entity-matching model. The entity-matching model finds the right vendor record. The Worth Score is a separate weighted multi-factor scoring system in manual-score-service.
🏦 Banking Category
Source: Plaid (platform 1)
Factors: avg daily balance, NSF count, overdraft frequency, deposit volume
Data: plaid_transactions
📋 Public Records
Source: Middesk, Trulioo, Adverse Media, NPI
Factors: SOS status, watchlist hits, adverse media, NPI validity, business age
Platforms: 16, 38, 27, 28
📊 Financial Strength
Source: Rutter (QBO, Xero, etc.) + Tax Status
Factors: revenue trends, gross profit, AR, tax compliance
Platforms: 5–13, 15
Integrations complete → Kafka: GENERATE_AI_SCORE → calculateBankingScore() + calculatePublicRecordsScore() + calculateFinancialScore() → Sum weighted_score_100 → × 8.5 → weighted_score_850 (0–850) → score_decision_matrix lookup: Low / Medium / High Risk + AUTO_APPROVED / REJECTED / MANUAL → UPDATE data_cases.status_id → UI shows "Low Risk ✓ 819/850 Generated on 03/27/26 using Model 3.0"

"Model 3.0" = score_config_history version referenced by business_scores.score_weightage_config

KYB Contact Information — Submitted vs. Reported Addresses

Address types

TypeMeaningSource
SubmittedApplicant typed this during onboardingdata_businesses.address_*
ReportedOfficial registry returned this addressbusiness_entity_address_source WHERE submitted=false

Verification badges

  • Verified Trulioo Comprehensive View: all address fields "match" or "missing"
  • No Match ANY address field returned "nomatch" in Comprehensive View
  • Deliverable USPS confirms mail deliverable (Middesk deliverable=true)
  • Google Profile: Verified SERP (platform 39) found matching Google Maps listing

Why multiple reported addresses?

Middesk and Trulioo query official registries that store multiple addresses for the same entity: registered office, trading address, mailing address, historical addresses. ALL are shown in the Contact Information tab — analysts can review any discrepancies.

Address storage

integration_data.business_entity_address_source: address_line_1, city, state, postal_code submitted = true (applicant entered) submitted = false (registry reported) deliverable = true/false (USPS check) integration_data.business_entity_review_task: key = "address_verification" status = "success" (Verified) | "failure" (No Match)

Match / No Match badges

These come from Trulioo PSC (UK/Canada) or Plaid IDV (US) — NOT the XGBoost entity-matching model.

Plaid IDV: applicant completes photo ID + selfie → Plaid compares submitted fields vs govt ID → Returns field-level match results → stored in: data_identity_verifications Trulioo PSC: name + DOB + address vs bureau/registry → DatasourceFields[{FieldName, Status}] → "match" → green Match badge → "nomatch" → red No Match badge → "missing" → no badge (field not in data)

Fraud Report — Plaid IDV Risk Intelligence

The Fraud Report section appears only when Plaid IDV is enabled for the customer. It comes from Plaid's risk_check which runs automatically during the identity verification flow.
UI FieldPlaid API fieldWhat it means
Synthetic Identity Risk Score risk_check.identity_abuse_signals.synthetic_identity.score 0–100: likelihood the identity was fabricated (real + fake info combined). Score 3 = very low. >70 = HIGH risk.
Stolen Identity Risk Score risk_check.identity_abuse_signals.stolen_identity.score 0–100: likelihood this is a real person's identity being used fraudulently. Score 1 = very low. >70 = HIGH risk.
Fraud Ring risk_check.fraud_ring_detected Signals coordinated fraud ring activity on this device/identity

Summary — The Admin Portal in One Page

What the portal shows you

  • All 249 customers, each with its own sandbox or production environment
  • Cases: point-in-time snapshots of onboarding or risk monitoring
  • KYB: vendor-enriched business facts — confidence is per-source, not per-address
  • KYC: owner self-entered data + IDV/PSC verification badges
  • Worth Score: weighted multi-factor model across banking, public records, financial strength

What it doesn't show

  • Vendor names (customers only see platform IDs)
  • Raw confidence scores from Trulioo watchlist hits
  • Auto-decisioning pass/fail rules (customer.joinworth.com only)
  • Pipeline B analytics data (Redshift internal only)
  • UK SIC / NACE codes (captured but not surfaced)
Fastest navigation tip: If you have a business ID, go directly to admin.joinworth.com/{business_id}. If you need vendor source names and confidence, use the Network tab or Postman to call GET /facts/business/{id}/details directly.
