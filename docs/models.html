Model 1 — Entity Matching XGBoost

How Worth AI identifies which vendor record belongs to each company

Model 1 — The Problem It Solves

When "Apple Inc" is submitted, vendor databases return hundreds of candidates. Which one is the real Apple Inc?

OpenCorporates: 200+ "Apple" records in California alone
"Apple Inc", "Apple Computer Inc", "Apple Bank for Savings", "Apple Valley USD"…
Equifax: Millions of US business records
Name variations, address changes, merged entities
ZoomInfo: 100M+ global company records
Subsidiaries, branches, rebranded companies
Liberty Data: Commercial intelligence overlay
Different name normalisation standards

The Model's Job

Input: "Apple Inc, 1 Infinite Loop, Cupertino CA"
↓ canonise + block candidates
Compute 26 similarity features per candidate pair
↓ XGBoost predict_proba
Apple Inc. (CIK 320193, CA) → 0.97 ✅ MATCHED
Apple Bank for Savings (NY) → 0.12 ❌
Apple Valley USD (CA) → 0.08 ❌

The Gap — Why Model 1 Alone Is Not Enough

What the deterministic rule cannot do, no matter how good the entity matching is

Worth AI After Entity Matching — A Deterministic Rule

The Rule
factWithHighestConfidence():
  winner = argmax(
    source.confidence × source.weight
  )
  if two_scores differ by < 0.05:
    use weightedFactSelector(
      fact_level_weight
    )

# Result: exactly ONE code, no probability
This rule was the right choice when Worth AI only needed to classify US companies with NAICS. It is fast, deterministic, and auditable.

It becomes insufficient as soon as you need probabilities, multi-taxonomy output, AML signals, or jurisdiction routing.

The 7 Structural Gaps

1. No probability output — always one winner, no uncertainty signal
2. Cannot learn — manual overrides in rel_business_industry_naics are ignored
3. Source conflict is silent — OC says Restaurant, Equifax says Holding Co → rule picks one, flags nothing
4. Agreement not aggregated — 5/5 sources agree = same format as 1/5
5. No jurisdiction routing — GB company still gets NAICS, not UK SIC
6. No temporal awareness — code changed 3x this year = normal classification
7. One taxonomy only — structurally impossible to output NAICS + UK SIC + MCC simultaneously

Why Calibrated Probability Changes Everything

Without probability (Worth AI today)

Company A: 5/5 sources agree → NAICS 334118
Company B: 1/5 sources respond → NAICS 334118

Both output: NAICS 334118 — no difference visible
Underwriter sees the same output for a well-matched Apple Inc and an unknown startup with one data point. Both get treated identically. Risk is invisible.

With calibrated probability (Consensus Engine)

Company A: P(NAICS 334118) = 0.92 → AUTO-APPROVE
Company B: P(NAICS 334118) = 0.14 → MANUAL REVIEW
Underwriters focus attention on the 14% confidence cases. High-confidence classifications flow through automatically. Risk is quantified and actionable.
What "calibrated" means: If the model says 0.90, roughly 90% of companies with similar features are correctly classified at that code. Not just a relative ranking — an absolute probability.

Model 2 — Classification Consensus XGBoost

The model Worth AI is missing — and why industry classification cannot be solved with a rule

Model 2 — Architecture and Setup

XGBoost Configuration
XGBClassifier(
  objective    = "multi:softprob",
  # → outputs probability per class
  # → all probs sum to 1.0 (softmax)

  num_class    = N,   # unique codes seen
  max_depth    = 5,   # tree complexity
  n_estimators = 80,  # boosting rounds
  learning_rate= 0.10,# shrinkage/round
  subsample    = 0.8, # row sampling
  colsample    = 0.8, # feature sampling
)
Why XGBoost and not a neural network?
• Handles mixed numeric features natively
• Works with small training sets (starts synthetic, grows with overrides)
• Calibrated probabilities out of the box with multi:softprob
• Sub-millisecond inference
• Interpretable via SHAP values

What multi:softprob means

For each company, the model outputs one probability per industry code, where all probabilities sum to 1.0:
probs = model.predict_proba(X)
# → [0.92, 0.04, 0.02, 0.01, 0.01, ...]
#    ^NAICS    ^541  ^519   ^551  ^423
#    334118   512   290    112   690

top_5 = argsort(probs)[::-1][:5]
# Primary:   334118 → 92%
# Secondary: 541512 → 4%
# Tertiary:  519290 → 2%
Training data: Synthetic multi-vendor samples covering all 200+ jurisdictions → replaced by real manual overrides from rel_business_industry_naics WHERE platform='manual'

Model 2 — Features Group A: Source Quality (12 features, idx 0–11)

These encode how reliable each of the 6 vendor signals is for THIS specific company. A source that matched at 97% confidence deserves far more weight than one that matched at 62%.

Model 2 — Features Group B: Semantic & AML Signals (5 features, idx 12–16)

Feature 12 — Trulioo Pollution Flag
1 if Trulioo returned a 4-digit SIC code in a jurisdiction that uses 5–6 digit codes. Known data quality issue — automatically detected and down-weighted.
Feature 13 — Web↔Registry Semantic Distance
Formula: cosine_distance(embed(registry_label), embed(ai_label))
Registry says: "Holding Company" (NAICS 551112)
AI infers from website: "restaurant food service"
→ Distance = 0.86 → SHELL COMPANY SIGNAL

>0.55 threshold → REGISTRY_DISCREPANCY risk flag
Feature 14 — Temporal Pivot Score
Formula: unique_codes / total_snapshots (3 quarters)

Stable restaurant: Q1=722511, Q2=722511, Q3=722511 → score=0.00
Suspicious pivot: Q1=722511, Q2=551112, Q3=541511 → score=1.00

>0.70 threshold → STRUCTURE_CHANGE AML flag (U-Turn fraud pattern)
Feature 15 — Cross-Taxonomy Agreement
Fraction of 6 taxonomy systems (NAICS/UK SIC/NACE/ISIC/SIC/MCC) that map to the same semantic cluster via UGO FAISS. High = high confidence. All 6 agree → near certainty.

Why 10 buckets instead of 200 columns?
200 one-hot columns for every country would create a sparse feature space. The 10 buckets capture the taxonomically meaningful distinctions (US→NAICS, EU/UK→SIC/NACE, global→ISIC) while keeping the model manageable.
This is the routing fix for Worth AI's core gap. Worth AI has jurisdiction_code in Redshift for 16M+ GB companies — it just never uses it to change the output taxonomy. Model 2 encodes this routing as learned feature weights. When feature 23 (EU) = 1, the model assigns higher probability to UK SIC and NACE codes. No hard-coded rules needed.

Model 2 — Features Group E: Agreement & Risk (7 features, idx 31–37)

Feature 31 — Majority Code Agreement
most_common_code_count / total_sources

5/5 agree on 334118 → 1.00 → HIGH confidence
All 5 different codes → 0.20 → LOW confidence

The rule cannot encode this. Whether 1 or 5 sources agree, the rule produces the same output format.
Feature 32 — High-Risk NAICS Prefix
1 if any source returns a code starting with AML-elevated 4-digit prefix:

5221/5222 — Banking intermediation
5511 — Holding companies
4236 — Electronic goods wholesale (dual-use)
9281 — Defense

Direct input to AML risk engine
Feature 33 — Unique Code Diversity
unique_codes / total_codes
High diversity = high disagreement = low confidence
Features 35–36 — Confidence Stats
Mean and max match confidence across all sources. Overall data quality indicator.
Feature 37 — Source Count
sources_returning_data / 6
Fewer sources = less reliable classification

Why Both Models Are Required — Not Alternatives

Without Model 1 (entity matching)
Cannot reliably find the right vendor record.
"Apple Inc" might match "Apple Bank for Savings"
→ Wrong industry code fed to Model 2

Features 0–11 (weighted confidences + MATCHED flags) depend entirely on Model 1's output. Without Model 1, Model 2 has no signal quality information.
Without Model 2 (consensus)
Finds the right records but cannot aggregate intelligently.
Source conflict → silent tie-breaking, not a flag.
No probability → no threshold-based auto-approval.
No jurisdiction routing → GB companies get NAICS.
No AML signals → fraud patterns invisible.

The Complete Two-Stage Pipeline

Model 1: "OC record is Apple Inc at 97% match confidence"
"Equifax record is Apple Inc at 91%"
"ZoomInfo at 94%, Liberty Data at 88%"
↓ build 38-feature vector from these confidences
+ jurisdiction (us) + entity type (Operating)
+ pivot score (0.00) + source agreement (4/5→334118)
Model 2: P(NAICS 334118) = 0.92 ← top-5 codes
Primary: NAICS 334118 · Secondary: UK SIC 26400
Risk: LOW · KYB: APPROVE
The result is greater than the sum of parts: Model 1 provides reliable evidence. Model 2 synthesises it into an actionable, multi-taxonomy, risk-annotated classification.

Summary


🟢 Model 1 (Entity Matching): Already in Worth AI. Answers "Is this the same company?" using 26 name+address features. Output feeds directly into Model 2 as weighted confidence features.
🔴 Worth AI classification: Not a model — a deterministic rule. Picks one NAICS winner. No probability. No learning. No jurisdiction routing. No AML signals.
🔵 Model 2 (Classification Consensus): New in Consensus Engine. 38 features encode ALL vendor signals, data quality, jurisdiction context, AML patterns. XGBoost multi:softprob outputs calibrated probability distribution → multi-taxonomy output + 9 risk signals.
🚀 Why both are needed: Model 1 finds the right evidence. Model 2 synthesises it into an actionable, probability-weighted, risk-annotated classification across all jurisdictions and taxonomies simultaneously.
