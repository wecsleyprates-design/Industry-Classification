How many facts

8

Worth AI runs two completely separate pipelines to answer this question. They share some inputs but have different goals, different source coverage, and different audiences. Understanding both is essential to understanding where the 8 facts come from.

Two Pipelines — Overview & Goals

🅐 Pipeline A — Integration Service

Goal: Deliver the best possible industry classification in real-time using ALL available sources.

Runs: Every time a business is submitted

Sources: ZoomInfo, Equifax, OC, Middesk, Trulioo, SERP, AI (6+ sources)

Output: rds_warehouse_public.facts (JSONB, 217 facts)

Confidence: XGBoost model (most sources)

✅ Customer sees this via REST API, Worth 360 Report, Worth AI UI

🅑 Pipeline B — Warehouse Service

Goal: Build a wide analytics table for model training, risk scoring, and bulk data exports.

Runs: Scheduled batch job in Redshift

Sources: ZoomInfo + Equifax ONLY

Output: datascience.customer_files (wide denorm. table)

Confidence: XGBoost entity matching model

⚠️ Internal only — not accessible to customers via API

⚡ The two pipelines can show different NAICS codes for the same business — Pipeline A uses all 6 sources while Pipeline B only compares ZoomInfo vs Equifax.

🅑 Pipeline B — Step 1

Heuristic Similarity Scoring

Before the XGBoost model runs, Worth AI finds candidate vendor records using Levenshtein string distance.

What similarity_index is

similarity_index =
  (20 - levenshtein(name_submitted, name_vendor))  ← max 20
+ (20 - levenshtein(addr_submitted, addr_vendor))  ← max 20
+ state_match   (0 or 1)
+ city_match    (0 or 1)
+ zipcode_match (0 or 1)
─────────────────────────────────
Max achievable: ~43–45

MAX_CONFIDENCE_INDEX = 55

Defined in sources.ts line 33. Normalises the raw score to 0–1:

confidence = similarity_index / 55

Index 45 → confidence 0.82 (just above match threshold)

Tables created

TableContent
smb_zoominfo_standardized_joinedTop 1000 ZI candidates per business ranked by similarity_index
smb_equifax_standardized_joinedTop 1000 EFX candidates ranked
smb_open_corporate_standardized_joinedTop 1000 OC candidates ranked

Matching threshold

similarity_index >= 45 (≈ 0.82 confidence) is the minimum to be considered a match via heuristic.
XGBoost threshold: probability >= 0.80

🅑 Pipeline B — Step 2

XGBoost Entity Matching Model

What the model answers

"Is this submitted business the same real-world entity as this vendor record?"

33 input features per pair

Jaccard k-grams on business name (k=1,2,3,4,word) · Street name k-grams · Short name (city-stripped) · Exact: city, ZIP, street number, address line 2 · Street number distance · Block-level match

Output table

datascience.ml_model_matches

The central XGBoost output table. One row per business per source.

ColumnMeaning
zi_probabilityProb. ZoomInfo record = same business (0–1)
efx_probabilityProb. Equifax record = same business (0–1)
oc_probabilityProb. OC record = same business (0–1)

Two-tier matching

XGBoost ≥ 0.80 → use XGBoost score directly
Else → use heuristic similarity_index / 55 if ≥ 45/55

🅑 Pipeline B — Step 3

How zi_match_confidence & efx_match_confidence Are Built

Source: smb_zi_oc_efx_combined.sql

-- ZoomInfo confidence:
CASE
  WHEN zi_probability IS NOT NULL
    THEN zi_probability          ← XGBoost score
  WHEN similarity_index/55.0 >= 0.8
    THEN similarity_index/55.0   ← heuristic fallback
  ELSE 0
END AS zi_match_confidence

-- Equifax confidence (same pattern):
CASE
  WHEN efx_probability IS NOT NULL
    THEN efx_probability
  WHEN similarity_index/55.0 >= 0.8
    THEN similarity_index/55.0
  ELSE 0
END AS efx_match_confidence

🅑 Pipeline B — Step 4

The Winner-Takes-All Rule (customer_table.sql)

-- Controls EVERY firmographic field in customer_files:
WHEN COALESCE(zi_match_confidence, 0) > COALESCE(efx_match_confidence, 0)
  THEN zi_c_naics6          -- ZoomInfo NAICS + ALL ZI firmographic data wins
ELSE efx_primnaicscode      -- Equifax NAICS + ALL EFX firmographic data wins

This one comparison controls: NAICS, employee count, revenue, company name, address, city, ZIP, country, website, affiliate flag, and more.

Important distinction — two different XGBoost scores

The XGBoost confidence shown in the API response (source.confidence) measures entity identity matching — "is this the right company?"
The zi_match_confidence and efx_match_confidence in Pipeline B are the same model's scores but stored differently for the batch Redshift pipeline.
Same model, different storage, different usage.

🅐 Pipeline A — Fact Engine

The 6 Winner Selection Rules

Rule 1 — factWithHighestConfidence()

Sort all sources by confidence. Highest wins outright if the gap between top two is > 5%.

|conf_A - conf_B| > 0.05 → higher wins

Rule 2 — weightedFactSelector() [tie-break]

If top two are within WEIGHT_THRESHOLD = 0.05:
Use source weight to break the tie.

Middesk(2.0) > OC(0.9) > ZI(0.8) = TRU(0.8) > EFX(0.7) > AI(0.1)

Rule 3 — manualOverride() [always wins]

Analyst sets via PATCH /facts/.../override/{factName}.
Stored with override: {…} field. Beats any model/AI result.

Rule 4 — No minimum confidence cutoff

There is no "confidence must be ≥ X" rule.
If only one source returned a code at confidence 0.05 — it wins.
Low confidence IS visible in the API response.

Rule 5 — AI safety net

AI enrichment triggers when fewer than 1 non-AI source returned a naics_code (minimumSources=1) AND total sources < maximumSources=3.
Weight = 0.1 so AI rarely wins unless vendors found nothing.

Rule 6 — Last resort: "561499"

If winning code not in core_naics_code table → removeNaicsCode() replaces it with 561499 (All Other Business Support Services).

🅐 Pipeline A — AI Enrichment

GPT-5-mini AI Enrichment — When & What

Trigger condition

DEPENDENT_FACTS.naics_code = {
  maximumSources: 3,   // SKIP if 3+ sources already
  minimumSources: 1,   // Need at least 1 for context
  ignoreSources: ["AINaicsEnrichment"]
}

What GPT receives

• Business name + address
• Website URL → GPT uses web_search tool to read it
• DBA names (if any)
• Existing codes from other sources (for correction context)

What GPT returns

{
  "naics_code":    "722511",
  "naics_description": "Full-Service Restaurants",

Post-processing validation

If not found → removeNaicsCode() → store "561499".

The 8 Facts — Known Gap

classification_codes — The Graveyard Fact

What it captures

Every industry code from every source and every taxonomy system — BEFORE the Fact Engine picks a winner.

{
  "codes": [
    { "system": "NAICS",
      "code": "722511",
      "source": "Middesk",
      "confidence": 0.95 },
    { "system": "uk_sic",
      "code": "56101",
      "source": "OC",
      "confidence": 0.89 },
    { "system": "nace",
      "code": "I5610",
      "source": "OC",
      "confidence": 0.89 }
  ]
}

The problem

No downstream consumer

UK SIC codes (e.g. gb_sic-56101) from OC are correctly captured here. But no Kafka handler, no API endpoint, no PDF report, no database column ever reads classification_codes. The UK SIC code exists in the system and is silently discarded.

What the Consensus Engine fixes

The Consensus XGBoost Level 2 model routes UK SIC as the primary output for GB companies — using the data that is already in classification_codes but currently ignored by the production pipeline.
